## 关键设计

### 多源发现 + 优雅降级

Steam 搜索节点依次尝试 4 个数据源：

1. **RSS** (最稳定) — 2 次重试，30s 超时
2. **热门新品 HTML** — 15s 超时
3. **Featured API** — 15s 超时
4. **即将推出 HTML** — 15s 超时

任一源成功即继续，全部失败返回空列表，管线不会中断。Epic 和 Tavily 管线独立运行。

### 并行迭代

Dify 迭代器配置 `parallel_nums=10`，最多 10 个游戏同时处理。每个游戏耗时约 10-30s（API 调用 + LLM），20 个游戏的总处理时间约 30-60s。

### SQL 生成与合并

LLM 节点为每个游戏生成两条 INSERT：
```sql
INSERT INTO games (...) VALUES (...);
INSERT INTO game_prices (...) VALUES (...);
```

执行节点用正则拆分为 games 和 prices 两组，各自合并为一条带 `ON DUPLICATE KEY UPDATE` 的批量 INSERT，减少数据库往返。

### 网络层

所有外部请求通过 Dify 沙箱 + SSRF 代理（Squid）链路：

```
Code Node (Python) → SSRF Proxy (Squid:3128) → Internet
```

SSL 证书验证已关闭（沙箱内网环境），超时分级设置（稳定源 30s，不稳定源 15s）。

## 数据表设计

### 去重策略

- **games**: `UNIQUE KEY (name, appid)` — 同名同 appid 视为同一游戏，更新其他字段
- **game_prices**: `UNIQUE KEY (appid, name, recorded_at)` — 每日每游戏一条价格记录

### 为什么 appid 可为 0？

Epic 游戏没有 Steam AppID，且 LLM 偶尔会为缺少 appid 的游戏输出 NULL。表结构用 `NOT NULL DEFAULT 0` 防御。

## 技术栈约束

- **仅 Python 标准库** — Dify 沙箱不支持 `pip install`
- **无外部 HTTP 库** — 只能用 `urllib.request`
- **无 lxml/BeautifulSoup** — HTML 解析只能用 `re`
- **XML 解析** — `xml.etree.ElementTree`（标准库）
- **SQL 执行** — 依赖 Dify SQL Execute 插件，每个节点只能执行一条 SQL
